# $\fbox{Chapter 1: DOMAIN DOCUMENTS}$





## **Topic - 1: Problem Domains**

### <u>Author Terminologies</u>

- **<u>Problem domain</u>:** Non-technical knowledge that software engineer needs to know before working on a project.
- **<u>Directly</u>:** Anything that software can directly control, like hardware.
- **<u>Indirectly</u>:** Anything that software doesn't control directly, like people using the software.


### <u>Problem Domains</u>

- One needs some electrical engineering knowledge to implement an OS.
- Its easier for electrical engineering students to implement an OS with some DSA knowledge.


### <u>Software-Exclusive Domains</u>

- These domains include software which are used by other software.
- **Examples -** Compilers, linkers, loaders, game engines, etc.



## **Topic - 2: Documents To Implement**

### <u>Introduction</u>

- There are two documents required to understand the problem domain.
- These are *software requirement document* and *software specification*.


### <u>Software Requirement Document</u>

- Contains all the information related to problem domain which might be useful for implementation.


### <u>Software Specification</u>

- Software specification document tells what outputs are expected for certain input.
- Or simply, its about how software's interface design, and not implementation details.
- Specification for hardware are more strict for being expensive and irreversible.
- It need not necessarily be done before implementation, or altogether.



## **Topic - 3: x86 Operating System**

### <u>Introduction</u>

- For software itself as the problem domain, the software requirement document and software specification may be present in single document.


### <u>Required Documents</u>

- *Intel® 64 and IA-32 Architectures Software Developer’s Manual (Volume 1, 2, 3)*
- *Intel® 3 Series Express Chipset Family Datasheet*
- *System V Application Binary Interface*


### <u>Intel Documents</u>

- *Intel* separates its software requirement document & software specification.
- Its software requirement document is known as *Functional Description*.
- For example, the *Register Description* section describes programming interfaces.
